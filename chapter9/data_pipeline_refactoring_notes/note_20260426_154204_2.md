# 我们已经完成了数据模型层的重构,测试覆盖率达到85%。下一步...

## 问题
我们已经完成了数据模型层的重构,测试覆盖率达到85%。下一步计划重构业务逻辑层。

## 分析
1. **结论**  
   建议在进入业务逻辑层重构前，先完成两项关键准备：**梳理并冻结数据模型层的公共接口**，以及**识别业务逻辑层与数据模型层之间的耦合模式**。直接重构业务逻辑层可能会因底层模型变动或接口不稳定而引入重复工作。

2. **依据**  
   - 历史笔记《数据模型层重构总结》中提到：“重构后所有模型对外暴露的API已标准化，但部分业务逻辑仍直接引用模型内部字段（如`user.name`而非`user.getDisplayName()`），需在下阶段解耦。”这表明业务逻辑层与数据模型层之间存在**紧耦合**，直接重构可能破坏已稳定的模型层。  
   - 笔记《测试覆盖率提升记录》指出：“数据模型层的85%覆盖率主要覆盖了CRUD操作和简单验证，**并未覆盖业务逻辑组合场景**。”因此，现有测试无法有效保护业务逻辑重构的正确性。  
   - 笔记《重构优先级清单》中标记了后续动作：“推荐先为业务逻辑层绘制**依赖关系图**，识别核心流程中的循环依赖或过度耦合点，再分模块重构。”

3. **风险与假设**  
   - **风险**：  
     - 如果业务逻辑层直接依赖数据模型层的内部结构（如私有属性或未文档化的API），重构时可能强制修改模型层，导致85%的测试用例失败，需回滚。  
     - 若未先定义业务逻辑层的接口契约，团队并行修改可能导致集成冲突。  
   - **假设**：  
     - 假设你已拥有业务逻辑层的单元测试框架（如pytest或unittest），且可以逐步引入测试替代重构（golden master testing）。  
     - 假设团队已就重构范围达成一致（例如：先重构用户认证模块，再重构订单处理模块）。

4. **下一步行动建议**  
   - **本周内**：  
     - 召开一次15分钟的设计评审会，确认数据模型层**公共接口**（如`UserRepository.find_by_id()`）的最终签名，并记录任何即将弃用的旧API。  
     - 从数据模型层测试套件中筛选出**集成测试**（至少20个用例），将其扩展为覆盖业务逻辑调用数据模型层的场景——这能作为重构前的基线。  
   - **两周内**：  
     - 使用静态分析工具（如pylint、mypy）或手动审查，为业务逻辑层绘制依赖图，标识出**超过3层调用链的方法**，优先解耦。  
     - 选择业务逻辑层中最“闭塞”的模块（如数据验证或权限检查）作为试点，用**依赖注入**替换直接引用模型类。  
   - **持续跟踪**：  
     - 在项目看板上创建一个“业务逻辑层重构-准备阶段”列，将以上任务拆为子项，并每两天同步进度。  
     - 如果发现任何阻断因素（如某个模型方法被20+业务逻辑调用），请立即标记为blocker，我会协助设计最小变更方案。
