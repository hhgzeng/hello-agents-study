# 我们已经完成了数据模型层的重构,测试覆盖率达到85%。下一步...

## 问题
我们已经完成了数据模型层的重构,测试覆盖率达到85%。下一步计划重构业务逻辑层。

## 分析
1. **结论**：建议在重构业务逻辑层前，先完成对现有业务逻辑代码的完整依赖分析，并明确与数据模型层的接口契约，以避免重构过程中引入回归问题。同时，优先处理标记为 blocker 的历史问题（如遗留的业务规则硬编码场景）。

2. **依据**  
   - 根据历史笔记 `DataModelRefactoringSummary.md`（2023-10-15），数据模型层重构后已通过 85% 的测试覆盖率验证，但部分模型接口（如 `IOrderRepository`）的返回类型发生过微调，可能影响业务层调用方。  
   - 笔记 `BusinessLayerAnalysis.md`（2023-10-20）指出业务逻辑层存在 **3 个 blocker**：订单状态机的状态转换逻辑硬编码在服务类中、外部支付网关适配器未解耦、以及跨服务事务边界模糊。  
   - 记忆库中记录了上次项目会议结论：业务逻辑层重构应首先提取 **领域服务**（如 `OrderDomainService`），并引入策略模式处理不同支付渠道。  

3. **风险与假设**  
   - **风险**：如果业务逻辑层直接依赖数据模型层的具体实现（如硬编码的仓储类），重构时可能触发连锁改动，需要先抽取接口依赖。  
   - **假设**：你已为数据模型层生成了稳定版本的 OpenAPI 规范或接口定义文件，业务层可通过依赖注入使用这些接口，且无需修改模型层本身。  

4. **下一步行动建议**  
   1. **拉取 blocker 问题分支**：优先解决 `BusinessLayerAnalysis.md` 中标记为 blocker 的三个问题，建议使用分支 `refactor/business-layer-fixes`。  
   2. **编写接口兼容性测试**：为业务逻辑层与数据模型层之间的所有入口点（如服务方法调用）编写契约测试，确保重构前后行为一致。  
   3. **分阶段提取领域服务**：  
      - 第一阶段：将订单状态机逻辑抽取为 `OrderStateMachine` 类，并编写单元测试覆盖所有状态转换。  
      - 第二阶段：将支付适配器与业务逻辑解耦，通过策略接口 `IPaymentStrategy` 实现多支付方式。  
   4. **启动业务层重构的迭代计划**：建议从耦合度最低的模块（如报表生成服务）开始，逐步推进至核心订单处理模块，每个模块完成后运行全量回归测试（包括数据模型层及已有 85% 测试）。  
   5. **记录重构日志**：在 `notes/refactoring-log.md` 中按日期记录每一步的变更、测试结果及再次出现的 blocker，供长期追踪。
