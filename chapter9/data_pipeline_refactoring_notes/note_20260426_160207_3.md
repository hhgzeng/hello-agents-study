# 我们已经完成了数据模型层的重构,测试覆盖率达到85%。下一步...

## 问题
我们已经完成了数据模型层的重构,测试覆盖率达到85%。下一步计划重构业务逻辑层。

## 分析
1. **结论**：  
   在开始业务逻辑层重构前，建议先完成接口审计与契约测试，确保业务逻辑层与已重构的数据模型层之间的依赖清晰可靠。同时利用现有85%的测试覆盖率作为安全网，采用增量替换策略而非一次性重写。

2. **依据**：  
   - 依据历史笔记中关于数据模型层重构的复盘（如“重构过程中因接口未明确导致回归问题”），明确接口一致性是跨层重构的关键风险点。  
   - 项目进展记录显示，测试覆盖率达到85%主要得益于数据模型层的单元测试和集成测试，这些测试覆盖了数据访问路径，但未必覆盖业务逻辑层的完整使用场景。  
   - 通用最佳实践：业务逻辑层重构时，若直接依赖数据模型层的实现细节（如内部缓存、查询优化），容易导致重构后语义偏离。因此需要先通过契约测试锁定接口行为。

3. **风险与假设**：  
   - **风险**：业务逻辑层可能存在对数据模型层“内部结构”的隐式依赖（如直接访问属性名、特定查询逻辑），这些依赖未被当前测试覆盖，重构时易被遗漏。  
   - **假设**：数据模型层已暴露稳定且文档化的公共API，且该API与业务逻辑层的现有调用方式一致（或可以通过适配器过渡）。如果实际接口有破坏性变更，需提前协调。

4. **下一步行动建议**：  
   - ① **审计依赖关系**：扫描业务逻辑层代码，列出所有对数据模型层的调用点和使用的数据类型，标记非公开API的引用。  
   - ② **编写接口契约测试**：为每个业务逻辑模块与数据模型层的接口编写契约测试（可使用Pact或类似工具），固化当前行为。  
   - ③ **制定增量计划**：将业务逻辑层划分为核心模块（如数据转换、规则引擎）和外围模块（如报表生成），优先重构核心且测试覆盖高的模块，逐步替换。  
   - ④ **引入适配层**：若发现直接依赖内部实现，先创建适配器或门面（Facade），隔离业务逻辑层与数据模型层的演进。  
   - ⑤ **执行回滚机制**：在每次重构后运行完整测试套件（包括数据模型层测试），确保85%覆盖率不下降，并标记可能退化的用例。
